# tsParticles 4.4.0 — Fluid in Collisions Mode Release Plan

## Status

4.4.0 fluid work is now defined as an extension of the existing collisions interaction.

This supersedes the obsolete standalone fluid package concept.

Primary technical source: `FLUID_INTERACTION_PLAN.md`.

---

## Release Feature Inventory

| # | Feature | Package | Type | Plan |
|---|---------|---------|------|------|
| H | Fluid particle behavior via collisions mode | `@tsparticles/interaction-particles-collisions` | Existing package extension | `FLUID_INTERACTION_PLAN.md` |

---

## High-Level Goal

Ship a stable `particles.collisions.mode = "fluid"` behavior that:

- uses DDR-based neighborhood resolution
- remains mutually exclusive with `bounce` / `absorb` / `destroy`
- requires no new plugin package or loader
- preserves backward compatibility for existing collision modes

---

## Scope Definition

## In scope

1. Extend collisions mode enum and options contract
2. Add fluid mode runtime branch inside collisions collider
3. Add documentation for collisions fluid mode and options
4. Add tests and regression checks for classic modes

## Out of scope

1. New package `@tsparticles/interaction-particles-fluid`
2. New loader `loadParticlesFluidInteraction`
3. Bundle wiring for a new fluid package
4. Engine architectural changes unrelated to collisions extension

---

## Implementation Phases

## Phase 0 — Baseline and guardrails

1. Validate current collisions package build + tests baseline
2. Record current behavior of `bounce`, `absorb`, `destroy` for regression reference
3. Confirm plan/docs no longer depend on standalone fluid package

Exit criteria:

- baseline green
- migration target clear (`collisions.mode = fluid`)

## Phase 1 — Contract extension (types + options)

1. Add `fluid` to `CollisionMode`
2. Define collisions fluid option interface/class
3. Wire `collisions.fluid` loading in collisions options class
4. Ensure defaults are present and safe

Exit criteria:

- options parse correctly
- type surface compiles

## Phase 2 — Runtime implementation

1. Extend collision particle runtime with fluid transient state
2. Implement pre-advection snapshot in `Collider.reset()` for fluid-mode particles
3. Implement fluid branch in `Collider.interact()`:
   - density and near-density accumulation
   - pressure computation
   - soft boundary spring
   - symmetric displacement
   - hard boundary clamp
   - velocity reconstruction
4. Keep classic collision branch untouched

Exit criteria:

- fluid mode runs without NaN/Infinity
- classic modes behavior unchanged by branch isolation

## Phase 3 — Stabilization and tuning

1. Validate behavior on representative configs (baseline/soft/dense)
2. Verify no mixed-mode artifacts
3. Tune defaults if needed for stability and visible cohesion
4. Check performance characteristics with medium/high particle counts

Exit criteria:

- stable visual behavior
- acceptable performance profile

## Phase 4 — Documentation and release readiness

1. Update collisions markdown docs (`mode` table + `collisions.fluid` options)
2. Ensure release notes describe fluid as collisions extension
3. Verify no docs mention standalone fluid package/loader as active path

Exit criteria:

- docs coherent with implementation
- release notes aligned with product decision

## Risks and Mitigation

### Risk 1: classic mode regression

Mitigation:

- strict mode branching
- targeted regression tests for `bounce`, `absorb`, `destroy`

### Risk 2: fluid instability (jitter/explosion)

Mitigation:

- enforce formula correctness (`dir * q * (P + Pnear * q) * 0.5`)
- force clamp
- distance epsilon guards
- conservative defaults

### Risk 3: boundary artifacts

Mitigation:

- soft spring + hard clamp sequence in fluid branch
- documented recommendation for `move.outModes`

### Risk 4: performance drop on dense scenes

Mitigation:

- neighbor cap
- grid-based queries
- avoid hot-loop allocations

## Acceptance Criteria

1. Fluid is configured through `particles.collisions.mode = "fluid"`.
2. Fluid tuning lives under `particles.collisions.fluid`.
3. No standalone fluid package/interactor/loader is introduced.
4. Fluid and classic collision modes are mutually exclusive at runtime.
5. Existing collision modes remain stable and backward compatible.
6. User-facing docs describe the collisions-based setup only.

2. Ship 4.4.0 without fluid mode
3. Preserve updated architectural decision (fluid-as-collisions-mode) for next iteration
