# tsParticles 4.3.0 — Master Coordination Plan

## Overview

This plan coordinates **3 independent features** targeting the `4.3.0` release.
Each feature has its own detailed handover plan. This document manages sequencing, dependencies,
conflict areas, and release readiness.

---

## Hard Rule: Master Plan First, Feature Plans After

All validation steps defined in **this** document (see "Step 0: Baseline Validation" below) **must be executed and passing before any feature-specific plan** (i.e. before `BACKGROUND_CANVAS_PLAN.md`, `FLUID_INTERACTION_PLAN.md`, `GIF_SHAPE_PLAN.md`) is worked on. Even if new feature plans are added to the 4.3.0 scope mid-cycle, run this master plan's checks first, then proceed to the feature plan's steps.

This rule exists because the 4.2.0 release was burned by a `workspace:*` → hardcoded version regression discovered only at CI publish time. Running the master gate first establishes a baseline so feature work doesn't mask or introduce regressions.

## Sequencing Strategy

Since all features are independent, the recommended execution is **parallel**:

```
Week 1-2:
  Agent C: GIF Shape (Steps 1-4)
  Agent A: Background Canvas (Phases 0-4)
  Agent B: Fluid Interaction (Steps 0-5)

Week 2-3:
  Agent C: GIF Shape (Steps 5-6)
  Agent A: Background Canvas (Phases 5-8)
  Agent B: Fluid Interaction (Steps 6-9)

Week 3-4:
  Integration: build all, integration tests, soak tests
  Release prep: changelog, docs, version bump to 4.3.0
```

If resources are limited, priority order:
1. **Feature C (GIF Shape)** — simplest, lowest risk, unblocks the image shape cleanup
2. **Feature A (Background Canvas)** — moderate, needs bundle-size measurement gate
3. **Feature B (Fluid Interaction)** — most complex, new simulation code, needs tuning

## Feature A — Background Canvas (from `BACKGROUND_CANVAS_PLAN.md`)

**Target:** `@tsparticles/engine`
**Load function:** N/A (built into engine)

**Design:** Multi-layer background pipeline (CSS → element auto-draw → draw callback → particles).
`element` and `draw` are independent layers; element is auto-composited via `ctx.drawImage()`,
`draw` always receives the main canvas context. See `BACKGROUND_CANVAS_PLAN.md` for full details.

### Key decisions (from plan)

1. **Bundle impact gate** (Phase 14): re-measure delta after v2 refactor; if LOW/MEDIUM → keep in engine; if HIGH → extract `@tsparticles/plugin-background`
2. **Plugin extraction**: if engine grows too large, `@tsparticles/plugin-background` owns all programmable layers; engine retains only CSS-style background

### Implementation phases (v2 refactor)

| Phase | Description | Verification |
|-------|-------------|-------------|
| 9 | Types/methods rename — widen element type (`HTMLVideoElement \| HTMLImageElement`), rename `#backgroundContext` → `#backgroundElement` | Compile passes |
| 10 | Rewrite `#drawBackground()` — independent element auto-draw (`ctx.drawImage`) + draw callback on main context | Layering order verified |
| 11 | Rewrite `#resolveBackgroundContext()` → `#resolveBackgroundElement()` — store element ref, no `getContext("2d")` | Element stored, context not extracted |
| 12 | Update tests — remove element-as-context tests, add auto-draw + layering tests | Tests green |
| 13 | Update docs — layers, video/image examples, worker+OffscreenCanvas example | Docs match design |
| 14 | Bundle impact re-measurement + plugin extraction decision | Size delta block |
| 15 | Validation gate | Final checklist complete |

### Critical files

| File | Change |
|------|--------|
| `engine/src/Options/Interfaces/Background/IBackground.ts` | Widen `element` type to include `HTMLVideoElement \| HTMLImageElement` |
| `engine/src/Options/Classes/Background/Background.ts` | Update property type |
| `engine/src/Core/RenderManager.ts` | `#backgroundContext` → `#backgroundElement`; rewrite `#drawBackground()`; rewrite `#resolveBackgroundElement()` |

## Rollback Plan

If any feature is not ready at the 4.3.0 freeze date:

| Scenario | Action |
|----------|--------|
| C not ready | Ship A (v2) + B as 4.3.0, defer C to 4.4.0 |
| A (v2) not ready | Ship C + B as 4.3.0, defer A to 4.4.0 (or ship A v1 as-is) |
| B not ready | Ship C + A (v2) as 4.3.0, defer B to 4.4.0 |
| C+A not ready | Ship B alone as 4.3.0 |
| C+A+B not ready | No 4.3.0 — reassess scope |

Each feature's branch must be independent so partial shipping is trivial.
